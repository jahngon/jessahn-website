# jessahn.com — personal website

Astro static site at https://jessahn.com. Personal site for Jess Ahn (GTM operator in SF, first-time coder). Visual direction: serif headlines, MUJI/Cuyana aesthetic, a hand-drawn blue squiggle accent.

## Audience and tone

- The owner is a first-time coder. When making changes, prefer the simpler path. Explain trade-offs in plain language before reaching for new abstractions.
- The site is read by a non-technical audience. Copy should be plain English, not jargon.

## Repo layout (what lives where)

- `src/pages/` — routes (`index.astro`, `artifacts/*`, `entries/*`, `about.astro`)
- `src/components/` — reusable Astro components. The big one is `CreditFreezeMatrix.astro` (the interactive widget on the child-credit-freeze-tool artifact page; ~1500 lines of Astro + plain TS + scoped CSS).
- `src/content/` — collections-driven content (entries)
- `public/widget/` — static files served same-origin to the credit-freeze widget:
  - `state.public.json` — bureau requirements + addresses (synced from the monitor repo)
  - `transunion-letter-bracketed.docx` / `.md` — letter templates (generated by `scripts/generate-transunion-template.mjs`)
- `scripts/` — one-off scripts:
  - `sync-monitor-data.mjs` — pulls latest `state.public.json` from the monitor repo into `public/widget/`
  - `generate-transunion-template.mjs` — regenerates the TransUnion .docx and .md from a single source of truth
- `STYLE_GUIDE.md` — design tokens, layout columns, typographic rules
- `.github/workflows/link-check.yml` — link check CI

## Common commands

- `npm run dev` — Astro dev server at http://localhost:4321
- `npm run build` / `npm run preview` — production build + local preview
- `npm run sync-monitor-data` — pull the latest monitor output into the website (run after the monitor records a change)
- `npm run generate-tu-template` — regenerate `.docx` + `.md` after editing the TransUnion letter content

## Cross-repo data flow

The credit-freeze widget gets its data from a separate repo:

```
child-credit-freeze-monitor (daily GH Action, 13:00 UTC)
        │
        │ writes data/state.public.json
        ▼
  (manual)  npm run sync-monitor-data
        │
        ▼
jessahn-website/public/widget/state.public.json  ← what the widget reads (same-origin)
```

The sync is currently manual. The widget is purely a reader — it never writes anywhere.

## Hard constraints — Child Credit Freeze Tool widget

The widget (`src/components/CreditFreezeMatrix.astro`) is purely informational. **It must never collect, store, or transmit any user information.** Specifically:

- No `<form>`, `<input>`, `<textarea>`, `<select>` elements
- No localStorage, sessionStorage, cookies, IndexedDB
- No backend, API, or analytics that touches anything a parent or child types
- No client-side PDF generation or form filling
- The only network call is `fetch('/widget/state.public.json')` — same-origin, read-only
- Mailing addresses are the single highest-stakes data point: a wrong address = a parent's freeze request goes to the void. The monitor alerts on address line changes; if an alert fires, treat it as a P0 until verified.
